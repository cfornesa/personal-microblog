## 2. Tables Safe To Drop

Each of the following is **completely absent** from the application source.
A repo-wide `rg` search for the table name, snake-case column references, and
likely camel-case wrappers returned zero hits in `artifacts/api-server`,
`artifacts/microblog`, `lib/api-spec`, `lib/api-zod`, and `scripts`.

| Table | Live rows | Notes |
|---|---|---|
| `categories` | 0 | No code references. |
| `post_categories` | 0 | Junction table for `categories` — drop with it. FK to `posts.id` and `categories.id`. |
| `feed_sources` | 0 | RSS-import feature that was never built into the active codebase. |
| `feed_items_seen` | 0 | Junction for `feed_sources` — drop together. |
| `nav_links` | 1 | No code reads or writes this; the React frontend hardcodes its nav. |
| `pages` | 1 | No CMS-pages route exists in `api-server`. The `@/pages/*` imports in `App.tsx` are React component files, not DB rows. FK from `nav_links.page_id` and `pages.author_user_id`. |
| `site_settings` | 1 | No code reads it. `feeds.ts` reads `SITE_TITLE` etc. from `process.env`, not from this table. |

### `reactions` — special case

- Defined in `lib/db/src/schema/reactions.ts` and exported from the package.
- **Not referenced by any route, handler, or frontend file.**
- 0 rows in the live DB.
- The README mentions reactions as a planned feature, but no endpoint reads or writes them.

**Recommendation:** treat as unused. Either drop it now, or keep the table but
remove the schema export to make the dead code obvious. I lean toward keeping
it because the schema file is real, the table is empty, and the README still
advertises the feature. Confirm before dropping.

### Suggested SQL (run only after a backup)

```sql
-- Drop in dependency order (children before parents).
SET FOREIGN_KEY_CHECKS = 0;

DROP TABLE IF EXISTS feed_items_seen;
DROP TABLE IF EXISTS post_categories;
DROP TABLE IF EXISTS categories;
DROP TABLE IF EXISTS nav_links;
DROP TABLE IF EXISTS pages;
DROP TABLE IF EXISTS site_settings;
DROP TABLE IF EXISTS feed_sources;

-- Optional, only if you're sure you don't want the planned reactions feature:
-- DROP TABLE IF EXISTS reactions;

SET FOREIGN_KEY_CHECKS = 1;
```

> Dropping `feed_sources` requires `posts.source_feed_id` to go first
> (see §3) because there is an FK `posts_source_feed_id_fk`.

## 3. Extra Columns On `posts`

Drizzle schema (`lib/db/src/schema/posts.ts`) defines exactly:
`id, author_id, author_user_id, author_name, author_image_url, content,
content_format, created_at`.

The live `posts` table has **five extra columns** that no code reads or writes:

| Column | Type | Live state | Reason it's safe to drop |
|---|---|---|---|
| `status` | `varchar(16)` NOT NULL, indexed | Likely all `'published'` | No `WHERE`, `INSERT`, or `UPDATE` references it. |
| `source_feed_id` | `int(11)` NULL, FK → `feed_sources.id` | All NULL | Pairs with the unused `feed_sources` table. |
| `source_guid` | `varchar(1024)` NULL | All NULL | RSS-import only. |
| `source_canonical_url` | `varchar(2048)` NULL | All NULL | RSS-import only. |
| `content_text` | `text` NULL, FULLTEXT index | NULL/empty | Search column for an unbuilt search feature. |

### Suggested SQL

```sql
-- Drop the FK and indexes first, then the columns.
ALTER TABLE posts DROP FOREIGN KEY posts_source_feed_id_fk;
ALTER TABLE posts DROP INDEX posts_source_feed_idx;
ALTER TABLE posts DROP INDEX posts_status_idx;
ALTER TABLE posts DROP INDEX posts_content_text_fulltext;

ALTER TABLE posts
  DROP COLUMN status,
  DROP COLUMN source_feed_id,
  DROP COLUMN source_guid,
  DROP COLUMN source_canonical_url,
  DROP COLUMN content_text;
```

> If you keep `feed_sources` (skipping §2), keep `source_feed_id` too.

## 4. Extra Columns On `users`

Drizzle schema (`lib/db/src/schema/users.ts`) defines:
`id, name, username, email, email_verified, image, bio, website, social_links,
role, status, created_at, updated_at, last_login_at, post_count`.

The live `users` table has **16 extra theming columns** that no code reads or writes:

```
theme, palette,
color_background, color_foreground,
color_background_dark, color_foreground_dark,
color_primary, color_primary_foreground,
color_secondary, color_secondary_foreground,
color_accent, color_accent_foreground,
color_muted, color_muted_foreground,
color_destructive, color_destructive_foreground
```

These appear identically on `site_settings` (which is itself unused). They
look like a per-user theming feature that was sketched into the DB but never
connected to the API or frontend. None of them appear in any `SELECT`,
`INSERT`, or `UPDATE` in the code.

### Suggested SQL

```sql
ALTER TABLE users
  DROP COLUMN theme,
  DROP COLUMN palette,
  DROP COLUMN color_background,
  DROP COLUMN color_foreground,
  DROP COLUMN color_background_dark,
  DROP COLUMN color_foreground_dark,
  DROP COLUMN color_primary,
  DROP COLUMN color_primary_foreground,
  DROP COLUMN color_secondary,
  DROP COLUMN color_secondary_foreground,
  DROP COLUMN color_accent,
  DROP COLUMN color_accent_foreground,
  DROP COLUMN color_muted,
  DROP COLUMN color_muted_foreground,
  DROP COLUMN color_destructive,
  DROP COLUMN color_destructive_foreground;
```

There's also a duplicate index on `username`: both `username` and
`users_username_unique` cover the same column. You can drop one:

```sql
ALTER TABLE users DROP INDEX username; -- keeps users_username_unique
```

## 5. Things That Look Suspicious But Are Actually Fine

- **`comments.author_id`** is `NOT NULL` while `author_user_id` is nullable.
  Both are still referenced in `comments.ts` (the legacy `author_id` is used as
  a fallback for ownership checks), so keep both.
- **`posts.author_id` / `posts.author_user_id`** — same story, both used.
- **`users.post_count`** — defined in code, but the API recomputes the post
  count on every read with `COUNT(*)`. Functionally redundant, but it's part
  of the declared schema and Drizzle still writes a default of `0`. Leave it
  unless you also want to remove it from `lib/db/src/schema/users.ts`.

## 6. Recommended Cleanup Order

1. **Back up the database.** `mysqldump` or your provider's snapshot.
2. Drop the extra `posts` columns (§3) — this also removes the FK that pins
   `feed_sources` in place.
3. Drop the unused tables (§2).
4. Drop the extra `users` columns (§4).
5. Optionally drop `reactions` (§2 special case) and `users.post_count`.
6. Run `npm run typecheck` and `npm run dev:api` to confirm the app still
   boots cleanly against the trimmed schema.
